# 开发日志 2026-04-06

## 难点突破总结

---

### 1. Worker 进程残留导致 LLM 推理全部崩溃

**问题：** 医疗问诊 Agent（medical_triage）连接后不说话、不响应用户语音。开场白（`session.say()`）正常播放，但之后的所有对话都静默。

**之前：** 无法从页面检测 Worker 状态，也无法重启。`create_room` 即使 dispatch 失败也静默继续，用户只看到"正在连接..."卡住。

**之后：**
- 页面增加 Worker 状态指示器（绿/红点）+ 刷新按钮 + 重启 Worker 按钮
- `create_room` 在 dispatch 失败时抛出 HTTP 503
- 连接后 15 秒无响应自动提示"Agent 未响应" + "断开并重试"按钮

**方法：**
1. 从 Worker 日志追踪到关键错误：`'SessionTracer' object has no attribute 'emit'`
2. 定位到 `backend/tracing/instrumented_llm.py`——该文件用 `self._ctx`（类型为 `TraceContext`，有 `emit()` 方法）调用 trace 事件
3. 但旧 Worker 进程还在运行，它的上下文是 `SessionTracer`（有 `_emit()` 而非 `emit()`）
4. 第一次修改把所有 `emit()` 改成 `_emit()` → 新 Worker 反而报错 `TraceContext has no attribute '_emit'`
5. 根因是**旧 Worker 进程没有干净退出**，代码本身是正确的（`TraceContext.emit()`）
6. 解决：改回 `emit()`，用 `pkill -9` 强杀旧进程，等 2 秒再启动新 Worker

**关键教训：** 排查问题时先确认是哪个进程在报错。两个 Worker 共存时，错误日志可能来自旧进程，误导修改方向。

### 2. Worker 进程管理——非优雅退出的级联问题

**问题：** `pkill` 发送 SIGTERM 后立即启动新 Worker，旧进程的 `aclose()` 还在执行，导致 `AssertionError`。两个 Worker 同时存在时抢占 LiveKit 房间。

**之前：** 只能手动到终端 kill 进程再重启，无页面入口。

**之后：** 页面增加 `/api/rooms/worker-restart` 端点，使用 `pkill -9`（SIGKILL）强杀 + 2 秒延迟后启动。Worker 状态通过 `list_rooms` API 轻量检测 LiveKit 连通性。

**方法：**
1. 最初用 `list_dispatch("")` 检查 Worker 状态 → 返回 404（房间不存在）
2. 改为 `list_rooms(ListRoomsRequest())`——轻量级连通性检查，无需具体房间名
3. 重启流程：`pkill -9 -f backend.pipeline.worker` → sleep 2s → `Popen` 启动新进程
4. 前端在 `IdleView` 中展示状态和操作按钮

---

### 3. TTS"无音频帧"错误的连锁反应

**问题：** CosyVoice TTS 报 `Please ensure input text is valid`，看起来像 TTS 问题。

**之前：** 误以为 TTS 配置或 CosyVoice 模型有问题，尝试修改 TTS 参数。

**之后：** 发现这是 LLM 崩溃的**下游症状**。LLM（`InstrumentedLLM`）在 trace emit 时异常退出，返回空文本给 TTS，TTS 自然报错。

**方法：**
1. 开场白正常 → 说明 TTS 链路本身没问题（`session.say()` 不经过 LLM）
2. 用户说话后无响应 → 问题出在 LLM 推理环节
3. 追踪到 `instrumented_llm.py` 中 `emit()` 方法的 AttributeError
4. 修复 LLM 层后 TTS 错误同步消失

**关键教训：** 音频管线中的错误容易误判定位层级。应从上游（ASR → LLM → TTS）依次排查，开场白正常 = VAD/ASR/TTS 链路没问题。

---

### 4. 对抗测试场景预填——提升测试效率

**问题：** 每次做 NLP 对抗测试都要手动输入场景描述（scenario），重复劳动且不一致。

**之前：** VoiceTestModal / ConversationTestModal 的场景字段为空，每次手动填。

**之后：** Agent 模型增加 `test_scenario` 字段，数据库 seed 为每个 Agent 预填专属场景。Modal 自动用 `defaultScenario={agent?.test_scenario || ""}` 初始化。

**方法：**
1. `backend/db/models.py` 增加 `test_scenario: Mapped[str | None]`
2. `backend/api/agents.py` 所有 CRUD 操作（Response/Create/Update/Duplicate/Snapshot/Rollback）加上该字段
3. `backend/db/seed.py` 为 11 个 Agent 编写针对性测试场景
4. 前端两个 Modal 组件新增 `defaultScenario` prop，用 `useState(defaultScenario)` 初始化

---

### 5. 双 VAD 冲突——Silero + DashScope server_vad

**问题：** 使用 DashScope Realtime ASR 时，同时启用了 Silero VAD 和 DashScope 自带的 server_vad，导致 VAD 判断冲突、语音截断异常。

**之前：** Silero VAD 和 DashScope server_vad 同时工作，两者对静音阈值/时长判断不一致，造成语音被提前截断或漏检。

**之后：** DashScope STT 场景下禁用 Silero VAD，只使用 DashScope 内置的 server_vad。

**方法：**
1. 在 `worker.py` 中检测 STT 类型，如果是 DashScope Realtime 则跳过 Silero VAD 初始化
2. DashScope server_vad 配置通过 `silence_duration_ms` 参数控制静音检测灵敏度
3. 保留 Silero VAD 作为非 DashScope ASR 场景的默认选项
